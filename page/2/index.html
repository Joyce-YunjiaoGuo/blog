二叉树是每个结点最多有两个子树的树结构。通常子树被称作“左子树”（left subtree）和“右子树”（right subtree）。

所谓遍历二叉树，就是按一定的规则和顺序走遍二叉树的所有结点，使每一个结点都被访问一次，而且只被访问一次。

设L、D、R分别表示遍历左子树、访问根结点和遍历右子树， 则对一棵二叉树的遍历有三种情况：DLR（称为先根次序遍历），LDR（称为中根次序遍历），LRD （称为后根次序遍历）。

前：a-b-d-e-f-g-c
中：d-e-b-g-f-a-c
后：e-d-g-f-b-c-a

前序的第一个一定为root!
中序里root左边为左子树所有节点的中序遍历，右边为右子树所有节点的中序遍历。

因为虚拟机JVM可以在内存上开辟一块空间，将 源文件（.java）编译成 字节码文件（.class），使其适用于不同的操作系统。

21.从输入url到页面加载完成到底发生了什么？
1、浏览器的地址栏输入URL并按下回车。
2、浏览器查找当前URL是否存在缓存，并比较缓存是否过期。（expire/max-age last-modified/etag）
3、DNS解析URL对应的IP。
4、根据IP建立TCP连接（三次握手）。
5、HTTP发起请求。
6、服务器处理请求，浏览器接收HTTP响应。
7、渲染页面，构建DOM树。
8、关闭TCP连接（四次挥手）。

22.请求报文 & 响应报文
请求报文：
a.请求行（请求方法,url,http协议版本）
b.请求头（user-agent/content-type/host/if-modified-since/if-none-match）
c.请求体（post数据）

响应报文：
a.响应行（http协议版本,状态码,状态描述）
b.响应头（server/content-type/last-modified/etag）
c.响应体（响应的数据）

23.浏览器缓存机制
请求头：if-none-match – 匹配 响应头中的etag 如果相同，则没有改变，304取缓存；如果不同，则200返回该资源和新的etag
请求头：if-modified-since – 匹配 响应头中的last-modified 如果相同，取缓存；不同返回新的资源

24.cookies vs localStorage sessionStorage vs sessions
cookies/localStorage/sessionStorage都保存在浏览器端
1.数据大小：cookie始终在请求中携带，在浏览器和服务器之间来回传递，所以小 4k
localStorage/sessionStorage仅在本地保存，大 5M
2.有效期：cookie过期前有效，expire/max-age
localStorage永久； sessionStorage仅在浏览器关闭前有效
3.作用域：sessionStorage不同窗口不共享
cookie/localStorage在所有同源窗口共享
session保存在服务器端
1.数据：cookie只能存字符串，session可以存任何数据类型
2.安全：cookie不安全，用户可以读到，session在服务器端较安全（账户密码）
3.有效期：cookie（expire/max-age）;session由服务器端决定
4.session太多，会消耗服务器性能

条件语句

if-else vs switch

if-else适用于判断两个离散值或几个不同的值域。
switch适用于判断多于两个离散值。

优化if-else

1.if-else中的条件语句应该按照从最大概率到最小概率排序，多数值直走一个语句。
2.嵌套的if-else语句–二分法
如果是1到10，则分为1-5，6-10，再往下分1-2，3-5；6-7，8-10。这样到达正确分支需要走的语句更少。

查找表

条件语句的数量很大的时候，查找表性能比if-else和switch好，可读性也更好。
单个键和单个值之间存在映射关系时，用查找表。

递归

递归可以把复杂的算法变得很简单，但终止条件不明确/缺少终止条件 => 函数长时间运行； “调用栈大小限制”。

调用栈溢出错误可以用try…catch捕获。异常类型因浏览器而不同。

如果不捕获它，会一直向上冒泡直到控制台。

递归一般有两种模式：
1.直接递归模式–函数调用自身。
2.隐伏模式–两个函数相互调用，形成一个无限循环。

导致栈溢出：
1.验证终止条件
2.太多层递归 => 改用迭代或memoization避免栈溢出

迭代

任何递归能实现的算法，同样可以用迭代来实现。
eg: 归并排序

memoization

memoization缓存前一个计算结果供后续计算使用，避免了重复工作。
eg: 阶乘

用JS操作DOM很耗性能！！！
？
因为浏览器通常把DOM和JS独立实现，如Chrome中，用Webkit中的WebCore来实现DOM和渲染，但JS部分是由独立的JS引擎–V8。
两个相互独立的功能只要通过接口彼此连接，就会产生性能消耗。所以用JS操作DOM次数越多，性能越差。
=>减少DOM操作，把运算尽量留在ECMAScript处理。

DOM访问与修改

访问DOM元素是有代价的，修改DOM元素代价更为昂贵。

最差！对HTML元素集合循环操作 每次循环操作，读取值一次，重写值一次

修改版本：一共只读取一次，重写一次

=> 减少DOM操作次数，尽量在ECMAScript中运算。

innerHTML vs DOM方法

绝大多数–旧版浏览器中，innerHTML性能更好；但在基于Webkit内核的新版浏览器中，DOM方法性能更好。

节点克隆

相比较创建新的元素document.createElement()， ele.cloneNode()效率更高一些。（先创建需要重复的元素，再clone）

HTML集合

HTML集合是包含了DOM节点引用的类数组对象。 以下返回值都为HTML集合：
document.getElementsByName(), document.getElementsByClassName(), document.getElementsByTagName()…
document.images, document.links, document.forms, document.forms[0].elements（页面中第一个表单的所有片段）…

底层DOM更新时，HTML集合也会自动更新 => 循环遍历时，每次迭代，都会重复执行查询
循环中访问length属性 解决方法：
1.将HTML集合复制到数组中，再处理这个数组。（大集合时，遍历数组快于遍历集合，但toArray中也多遍历了一次集合）

2.缓存length（遍历较小集合时）

循环中访问元素集合 解决方法：
使用局部变量存储集合引用 el

遍历DOM

非递归方式遍历元素所有子节点：用nextSibling来获取每个相邻元素 或 用childNodes得到子元素集合。

DOM自带属性不区分元素节点，注释节点，文本节点=>多DOM操作过滤元素节点。所以，可以使用 只返回元素节点的API:
children - childNodes
childElementCount - childNodes.length
firstElementChild - firstChild
lastElementChild - lastChild
IE只支持children， 其他不支持。所有浏览器中，children都比childNodes快。

使用 css选择器，查找元素更快，返回匹配节点的类数组对象，而不是HTML集合，避免可能的性能问题。

querySelector()获取第一个匹配的节点。

重排和重绘

浏览器下载完页面中的所有资源后，会解析并生成两个内部数据结构–DOM树和css树 => render树

几何属性变化（添加或删除可见的DOM元素，元素位置改变，尺寸改变，内容改变，页面渲染器初始化，浏览器窗口尺寸改变，滚动条） => 重排

由于每次重排都会产生计算消耗，大多数浏览器通过队列化修改并批量执行来优化重排过程。
但以下方法会强制刷新队列并立即执行：
offsetTop, offsetLeft, offsetWidth, offsetHeight
scrollTop, scrollLeft, scrollWidth, scrollHeight
clientTop, clientLeft, clientWidth, clientHeight
getComputedStyle(), currentStyle() in IE
！！！这些方法最好集中到最后一起执行，避免多次强制执行消耗性能。
