21.从输入url到页面加载完成到底发生了什么？
1、浏览器的地址栏输入URL并按下回车。
2、浏览器查找当前URL是否存在缓存，并比较缓存是否过期。（expire/max-age last-modified/etag）
3、DNS解析URL对应的IP。
4、根据IP建立TCP连接（三次握手）。
5、HTTP发起请求。
6、服务器处理请求，浏览器接收HTTP响应。
7、渲染页面，构建DOM树。
8、关闭TCP连接（四次挥手）。

24.cookies vs localStorage sessionStorage vs sessions
cookies/localStorage/sessionStorage都保存在浏览器端
1.数据大小：cookie始终在请求中携带，在浏览器和服务器之间来回传递，所以小 4k
localStorage/sessionStorage仅在本地保存，大 5M
2.有效期：cookie过期前有效，expire/max-age
localStorage永久； sessionStorage仅在浏览器关闭前有效
3.作用域：sessionStorage不同窗口不共享
cookie/localStorage在所有同源窗口共享
session保存在服务器端
1.数据：cookie只能存字符串，session可以存任何数据类型
2.安全：cookie不安全，用户可以读到，session在服务器端较安全（账户密码）
3.有效期：cookie（expire/max-age）;session由服务器端决定
4.session太多，会消耗服务器性能

条件语句

if-else vs switch

if-else适用于判断两个离散值或几个不同的值域。
switch适用于判断多于两个离散值。

优化if-else

1.if-else中的条件语句应该按照从最大概率到最小概率排序，多数值直走一个语句。
2.嵌套的if-else语句–二分法
如果是1到10，则分为1-5，6-10，再往下分1-2，3-5；6-7，8-10。这样到达正确分支需要走的语句更少。

查找表

条件语句的数量很大的时候，查找表性能比if-else和switch好，可读性也更好。
单个键和单个值之间存在映射关系时，用查找表。

递归

递归可以把复杂的算法变得很简单，但终止条件不明确/缺少终止条件 => 函数长时间运行； “调用栈大小限制”。

调用栈溢出错误可以用try…catch捕获。异常类型因浏览器而不同。

如果不捕获它，会一直向上冒泡直到控制台。

递归一般有两种模式：
1.直接递归模式–函数调用自身。
2.隐伏模式–两个函数相互调用，形成一个无限循环。

导致栈溢出：
1.验证终止条件
2.太多层递归 => 改用迭代或memoization避免栈溢出

迭代

任何递归能实现的算法，同样可以用迭代来实现。
eg: 归并排序

memoization

memoization缓存前一个计算结果供后续计算使用，避免了重复工作。
eg: 阶乘

因为浏览器通常把DOM和JS独立实现，如Chrome中，用Webkit中的WebCore来实现DOM和渲染，但JS部分是由独立的JS引擎–V8。
两个相互独立的功能只要通过接口彼此连接，就会产生性能消耗。所以用JS操作DOM次数越多，性能越差。

高性能javascript2--数据存取

发表于 2019-03-05 | 更新于 2019-03-06

数据存储的位置 => 数据读写性能

4中基本的数据存取位置：
1.字面量：字符串、数字、布尔值、对象、数组、函数、正则、null、undefined
2.本地变量：使用var定义的数据存储单元
3.数组元素：存储在JS数组对象内部，以数字为索引
4.对象成员：存储在JS对象内部，以字符串为索引

存取数据时间： 字面量 = 本地变量 < 数组 = 对象
多用字面量和局部变量，减少数组项和对象成员的使用

管理作用域

作用域链

每一个js函数都表示为一个对象，是Function对象的一个实例。Function对象有一个内部属性(只有js引擎可存取))[[scope]]。
[[scope]]包含了一个函数被创建的作用域中对象的集合–作用域链。

当函数被创建时 => GO全局对象
当函数被执行时 => 执行期上下文 函数每次执行时对应的执行环境都是独一无二的，因此多次调用同一个函数就会导致创建多个执行期上下文。当函数执行完毕，执行期上下文就被销毁
AO 作用域链最顶端，当执行期上下文被销毁，AO也被销毁

！！！读取数据性能优化
函数执行时，变量解析过程： 从作用域顶端即当前函数的AO开始，直到GO。如果没找到，则为undefined。如果两个位于作用域链不同部分的同名变量，则会返回第一个找到的值。
变量所在的位置越深，读写速度越慢。因此读取局部变量快，读取全局变量慢 => !!!尽可能使用局部变量，多次使用的全局变量的引用可以存储到一个局部变量里

改变作用域链

一般来说，一个执行环境的作用域链是不会改变的，除了with, try…catch…和eval()。=>动态作用域尽量少用

with用来给对象的所有属性创建一个变量，避免多次书写，但会产生性能问题。

当执行到with时，一个新的对象变量（包含document对象所有属性）被置于作用链的头部，意味着函数的局部变量处于作用链的第二个对象中。
这使得访问document对象的属性非常快，但访问局部变量较慢。=> !!!最好避免使用with

try…catch…用于捕捉错误，也会改变作用域链，产生相同的性能问题。

当try中的代码块发生错误，执行过程会自动跳转到catch子句，然后把异常对象推入一个变量对象并位于作用域链的首位，局部变量则变为第二个。
一旦catch执行完毕，作用域链就会恢复之前状态。
=> 为减小其对性能的影响，可以将错误委托给一个函数来处理。（只执行一条语句&&没有局部变量的访问）

闭包

闭包：允许函数访问局部作用域之外的数据，但可能导致性能问题。

！！！
1.闭包会拿到当前函数的AO和GO，当函数调用完，自己的AO和执行期上下文一起销毁，但作用域链的引用仍保存在了闭包中，无法销毁=>更大的内存开销=>内存泄漏

2.当闭包代码执行时，会创建一个执行期上下文和一个放在作用域链最顶端的AO，于是在访问assignEvents的AO和GO时，每次访问都带来性能损失。
！！！可以将常用的跨作用域变量存储在局部变量中，直接访问局部变量。

对象成员

对象成员包括属性和方法。 ？为啥访问对象慢呢

原型

JS中的对象是基于原型的。对象通过proto属性绑定到其原型，默认每个对象obj.proto = Object;

原型链

book1.proto = Book.prototype
Book.prototype.proto = Object
当调用原型链上的方法时，访问的位置越深，找到的越慢，每深入一层原型链都会增加性能损失。

嵌套成员

每次遇到点操作符，潜逃成员会导致js引擎遍历素有对象成员，对象成员嵌套越深，读取速度越慢。
location.href比window.location.href要快
如果不是对象自己的属性，还要去原型链上找，花更多时间。

！！！=> 缓存对象成员值
在函数中如果要多次读取同一个对象属性，最佳做法时将属性值保存到局部变量中，减少性能开销。但要注意this指向问题！

javascript的阻塞特性

当浏览器再执行js代码时， 不能同时做其他事情。页面不能下载和渲染，因为js会改变页面内容。

为了解决这个问题， 主要有以下几种方法：

1.将脚本放在底部

如果把大量js文件放在head中，js文件的下载和执行会阻塞页面加载，导致明显的延迟，通常表现为空白页面，用户无法浏览和交互。
IE8,Firefox3.5,Safari4和Chrome2都允许并行下载js文件，js文件间不阻塞，但仍阻塞其他资源的下载。
于是，将所有script标签尽可能放到body标签的底部，尽量减少对整个页面下载的影响。等到下载执行底部的js时，大部分页面已经下载并显示给用户。

2.将js文件合并成一个再引入

由于每一个script标签下载时都会阻塞页面渲染，所以要减少页面包含的script标签数量，尤其是外链脚本。
文件合并：离线打包工具

3.无阻塞脚本

仅仅减少js文件大小和限制http请求数，不能彻底解决问题，于是–无阻塞脚本–在页面加载完成后才加载js代码。

a.延迟的脚本

defer: 页面解析到script标签后开始下载对应js文件，与其他资源并行下载，直到dom加载完成，在window.onload事件触发前，执行。 (html4)
async: 页面解析到script标签后开始下载对应js文件，与其他资源并行下载，下载完后自动执行。（html5）

！！！两者都浏览器兼容性不好

b.动态脚本元素

js文件在script被添加到页面时开始下载， 下载完立即执行。无论何时启动下载，下载和执行过程不会阻塞页面其他进程

动态加载js文件的封装函数

！！！1. 兼容好 2.易用 => 最常用

c.XMLHttpRequest脚本注入

1.下载js但不立即执行，script添加到页面中才执行 2.兼容性好
！！！局限性：js文件和所请求页面必须处于相同的域，因此大型web应用通常不会采用XHR脚本注入
