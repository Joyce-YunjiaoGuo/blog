react-PureComponent

发表于 2019-11-18 | 更新于 2019-11-19

PureComponent

一个组件的state或props改变后 => render => 其后代都会重新render，即使没有state或props更新
PureComponent就是为了这种不必要的render，提升效率

原理就是在shouldComponentUpdate中优化，浅比较props和state是否改变，改变了重新render。

tips!!!

  1. 浅比较：只比较对象的第一层属性。 引用值比较其地址。
    所以，setState时，用新的对象/数组覆盖，而不是直接改变原来的

  2. 函数组件，没有办法extends PureComponent; 用React.memo()

React.memo()本质上就是给函数组件Task套了一层extends PureComponent的类组件

react-context上下文

发表于 2019-11-17

Context上下文

为了解决跨层级共享数据，否则需要层层用props传递，太麻烦！

组件App创建了一个context，则其后代都可以共享这个context数据。
但对于其后代而言，数据来源不仅仅是state和props，还有可能是context！有隐患！

旧版context（v16.0.0之前版本）

  • 如何创建上下文：

    1. 必须写childContextTypes类型约束
    2. getChildContext(), 在render之后执行，返回的对象即为context，其必须满足类型约束

  • 后代如何获取上下文

    1. 必须写contextTypes类型约束
    2. this.context.xxx或context.xxx取值

改变context值

  • 组件想要改变自己的context， 必须通过setState。
    把需要改变的context值放入state，setState改变后，组件会重新render，执行getChildContext，context中值随之更新。

  • 后代想要改变context，需要通过context中函数调用

有多个父级context情况下，子组件取离自己最近父级的context

ChildA中，this.context.a取123
ChildB中，this.context.a取789

新版（v16.0.0以后版本）

  • 如何创建上下文

  • 后代如何获取上下文

ctx与生成组件最好一一对应，需要新的上下文时，可以重新createContext生成一个！

强制更新

即使setState({})空对象，Provider的value值更新为一个新的对象，地址改变，其后代必须重新render，无论shouldComponetUpdate返回true or false

深入认识setState

发表于 2019-11-11

setState可能是异步的

可能： 当在html元素的事件中，setState是异步的；
其他都是同步的。

在实际开发中，把所有setState都当成异步。

同步调用多次setState, 需要用setState的函数形式，得到最新状态再改变

this.setState(prevState => {});

React会对异步的setState优化，合并操作。全部更新后，再执行render

因为 操作dom消耗性能

this.setState(pre => {
n: pre.n + 1,
}, () => console.log(this.state.n));

this.setState(pre => {
n: pre.n + 1,
});

this.setState(pre => {
n: pre.n + 1,
})

将3次setState放入队列中，
逐一改变，
全部完成后，
render,
执行回调console.log(this.state.n) — 3

vim的最基本使用

发表于 2019-10-26

最近公司的代码库，突然要求强制加hook，然后发现每次commit代码的message必须以指定代码开头，不然就无法push到远程。
一直报错： commit id cannot be empty。
但是！有一次填写commit message时，我加了-，就报错！于是我只能修改这一次的message，再重新push。
以前从来没这样操作过，于是，我去百度了一下：

git commit –amend 可以修改最近一次commit message

然而！进入了我不太会的vi界面！于是就有了这篇文章！

三种模式

基本上 vi/vim 共分为三种模式，分别是
命令模式（Command mode），
输入模式（Insert mode），
底线命令模式（Last line mode）。

这三种模式的作用分别是：

命令模式：

用户刚刚启动 vi/vim，便进入了命令模式。
此状态下敲击键盘动作会被Vim识别为命令，而非输入字符。

以下是常用的几个命令：

  • i 切换到输入模式，以输入字符。
  • x 删除当前光标所在处的字符。
  • : 切换到底线命令模式，以在最底一行输入命令。

命令模式只有一些最基本的命令，因此仍要依靠底线命令模式输入更多命令。

输入模式：

在命令模式下按下i就进入了输入模式。

在输入模式中，可以使用以下按键：

  • 字符按键以及Shift组合，输入字符
  • ENTER，回车键，换行
  • BACK SPACE，退格键，删除光标前一个字符
  • DEL，删除键，删除光标后一个字符
  • 方向键，在文本中移动光标
  • HOME/END，移动光标到行首/行尾
  • Page Up/Page Down，上/下翻页
  • Insert，切换光标为输入/替换模式，光标将变成竖线/下划线
  • ESC，退出输入模式，切换到命令模式

底线命令模式

在命令模式下按下:（英文冒号）就进入了底线命令模式。
在输入模式下按下shift + : 就进入了底线命令模式。

底线命令模式可以输入单个或多个字符的命令，可用的命令非常多。

在底线命令模式中，基本的命令有（已经省略了冒号）：

  • q 退出程序
  • w 保存文件
  • 按ESC键可随时退出底线命令模式

于是！想要修改最近一次commit message时，使用git commit –amend 进入vim后：

命令模式–按i进入输入模式–>
此时可以修改第一行的commit message–修改完成后，按shift+i进入底线命令模式–>
此时光标来到最后一行–输入wq!–>强制保存并退出

最近一次的commit message就修改完成啦~

这次操作只用了非常简单，为了以防万一，贴上vim比较全的操作用法。
